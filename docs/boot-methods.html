Understanding Boot Methods

This guide explains different methods to boot PowerVM LPARs for OpenShift SNO installation. Each method has different requirements for DHCP/BOOTP, automation level, and complexity. Choose the method that best fits your environment and requirements.

Available Boot Methods

Method 1: Network Boot with Minimal BOOTP RECOMMENDED

Description: Automated network boot using lpar_netboot with minimal BOOTP support. This is the default method used by this automation tool.

Advantages

  • Fully automated
  • Minimal DHCP usage (1-minute lease)
  • Scalable for multiple LPARs
  • Fast deployment
  • Easy to maintain
  • Works with existing workflow

Disadvantages

  • Requires minimal BOOTP/DHCP
  • Needs dnsmasq service
  • Network dependency
Implementation Status: This method is fully implemented in the automation. Set dhcp.enabled: false in your configuration file.

How It Works

  1. HMC initiates network boot via lpar_netboot command
  2. LPAR sends BOOTP request
  3. dnsmasq responds with minimal BOOTP reply (1-minute lease)
  4. LPAR downloads GRUB bootloader from TFTP
  5. GRUB reads configuration with static IP kernel parameters
  6. Kernel boots with static IP (no DHCP lease used)

Configuration

# In my-vars.yaml
dhcp:
  router: "192.168.79.2"
  netmask: "255.255.255.0"
  enabled: false  # Minimal BOOTP mode

sno:
  ipaddr: "192.168.79.10"
  macaddr: "fa:4e:86:23:37:20"

Command Used

lpar_netboot -i -f -t ent -m fa4e86233720 \
  -s auto -d auto \
  -S 192.168.79.2 \
  -C 192.168.79.10 \
  -G 192.168.79.2 \
  -K 255.255.255.0 \
  sno default_profile Server-9080-HEX-SN785EDA8

Method 2: ISO-Based Installation ADVANCED

Description: Boot from a custom ISO image with embedded ignition configuration. No network boot required.

Advantages

  • No BOOTP/DHCP required
  • No network boot needed
  • Complete control
  • Works in isolated environments
  • Portable installation media

Disadvantages

  • Manual ISO creation
  • ISO management overhead
  • Slower than network boot
  • Requires virtual media or HMC storage
  • Less automated
Best For: Completely isolated networks, environments without TFTP access, one-time installations, or air-gapped deployments.

Implementation Steps

  1. Download RHCOS live ISO
  2. Generate ignition configuration
  3. Embed ignition into ISO using coreos-installer
  4. Upload ISO to HMC or storage location
  5. Create virtual optical device on HMC
  6. Attach ISO to LPAR
  7. Boot LPAR from ISO
  8. Monitor installation
  9. Detach ISO after completion

Commands

# 1. Create custom ISO with embedded ignition
coreos-installer iso ignition embed \
  -i /path/to/sno.ign \
  -o custom-rhcos-sno.iso \
  rhcos-live-ppc64le.iso

# 2. Upload to HMC (example using scp)
scp custom-rhcos-sno.iso hmc_user@hmc:/var/vopt/

# 3. Create virtual optical device on HMC
mkvopt -name sno-install \
  -file /var/vopt/custom-rhcos-sno.iso \
  -ro

# 4. Assign to LPAR
chhwres -r virtualio --rsubtype vopt \
  -m  -o a \
  -p  \
  -s 1 -a "adapter_type=server,remote_lpar_id=1,remote_slot_num=1"

# 5. Boot from virtual optical
chsysstate -r lpar -m  -o on \
  -n  -f  \
  -b of

Method 3: Direct LPAR Profile Boot ADVANCED

Description: Configure network boot settings directly in the LPAR profile. Boot without lpar_netboot command.

Advantages

  • No BOOTP/DHCP required
  • Persistent configuration
  • Faster subsequent boots
  • Network settings in profile
  • No command-line boot needed

Disadvantages

  • Complex initial setup
  • Requires HMC profile modification
  • Profile changes per LPAR
  • Less flexible
  • Harder to automate
Best For: Permanent LPAR configurations, environments with few LPARs, situations where network boot settings rarely change.

Implementation Steps

  1. Access HMC command line or GUI
  2. Locate LPAR profile to modify
  3. Add network boot configuration
  4. Set boot device order (network first)
  5. Configure static IP parameters
  6. Set TFTP server and boot file
  7. Save profile
  8. Activate LPAR with modified profile

HMC Commands

# 1. View current profile
lssyscfg -r prof -m  \
  --filter "lpar_names=" \
  -F name,boot_mode

# 2. Modify profile with network boot settings
chsyscfg -r prof -m  -i \
  "name=,\
   lpar_name=,\
   boot_mode=norm,\
   net_boot_device=ent0,\
   net_boot_server=192.168.79.2,\
   net_boot_file=boot/grub2/powerpc-ieee1275/core.elf,\
   net_boot_client_ip=192.168.79.10,\
   net_boot_gateway=192.168.79.2,\
   net_boot_subnet_mask=255.255.255.0"

# 3. Activate LPAR with profile
chsysstate -r lpar -m  -o on \
  -n  -f

Method 4: Manual SMS Configuration MANUAL

Description: Manually configure network boot through the System Management Services (SMS) menu during LPAR boot.

Advantages

  • No automation required
  • Full interactive control
  • Good for troubleshooting
  • No BOOTP/DHCP needed
  • Can verify settings visually

Disadvantages

  • Completely manual
  • Not scalable
  • Time-consuming
  • Requires console access
  • Error-prone
  • Not repeatable
Best For: One-time installations, troubleshooting boot issues, testing configurations, learning the boot process.

Implementation Steps

  1. Boot LPAR and press F1 to enter SMS menu
  2. Select "Setup Remote IPL (Network Boot)"
  3. Choose network adapter
  4. Select "IP Parameters"
  5. Enter Client IP Address: 192.168.79.10
  6. Enter Server IP Address: 192.168.79.2
  7. Enter Gateway IP: 192.168.79.2
  8. Enter Subnet Mask: 255.255.255.0
  9. Enter Boot File: boot/grub2/powerpc-ieee1275/core.elf
  10. Save configuration
  11. Exit SMS and boot

SMS Menu Navigation

SMS Menu Structure:
├── Boot Options
│   └── Setup Remote IPL (Network Boot)
│       ├── Select Network Adapter
│       │   └── ent0 (or your adapter)
│       ├── IP Parameters
│       │   ├── Client IP Address: 192.168.79.10
│       │   ├── Server IP Address: 192.168.79.2
│       │   ├── Gateway IP: 192.168.79.2
│       │   └── Subnet Mask: 255.255.255.0
│       ├── Boot File
│       │   └── boot/grub2/powerpc-ieee1275/core.elf
│       └── Save and Exit
└── Exit SMS and Continue Boot

Method 5: Custom Boot Script ADVANCED

Description: Create a custom script that configures network and initiates boot without BOOTP.

Advantages

  • Highly customizable
  • No BOOTP required
  • Can include custom logic
  • Reusable once created
  • Automated execution

Disadvantages

  • Requires script development
  • Complex to implement
  • Platform-specific
  • Harder to maintain
  • Need to inject into boot process
Best For: Custom requirements, advanced automation needs, environments with specific boot workflows, integration with existing tools.

Example Implementation

#!/bin/bash
# custom-network-boot.sh
# Custom network boot script for PowerVM LPAR

set -e

# Configuration
CLIENT_IP="192.168.79.10"
NETMASK="255.255.255.0"
GATEWAY="192.168.79.2"
TFTP_SERVER="192.168.79.2"
INTERFACE="eth0"
BOOT_FILE="boot/grub2/powerpc-ieee1275/core.elf"

# Configure network interface
echo "Configuring network interface ${INTERFACE}..."
ip addr add ${CLIENT_IP}/${NETMASK} dev ${INTERFACE}
ip link set ${INTERFACE} up
ip route add default via ${GATEWAY}

# Verify network connectivity
echo "Testing connectivity to TFTP server..."
ping -c 3 ${TFTP_SERVER}

# Mount TFTP root via NFS
echo "Mounting TFTP root..."
mkdir -p /mnt/tftp
mount -t nfs ${TFTP_SERVER}:/var/lib/tftpboot /mnt/tftp

# Execute bootloader
echo "Executing bootloader..."
/mnt/tftp/${BOOT_FILE}

# Cleanup on failure
trap 'umount /mnt/tftp 2>/dev/null' EXIT

Integration with Automation

# Ansible task to deploy custom boot script
- name: Deploy custom boot script
  copy:
    src: custom-network-boot.sh
    dest: /var/lib/tftpboot/scripts/boot.sh
    mode: '0755'

- name: Configure LPAR to use custom script
  shell: |
    ssh {{ pvm_hmc }} "chsyscfg -r prof -m {{ sno.pvmcec }} -i \
      'name={{ profile }},\
       lpar_name={{ sno.pvmlpar }},\
       boot_mode=norm,\
       boot_script=/scripts/boot.sh'"

Method Comparison

Method BOOTP Required Automation Complexity Scalability Best Use Case
1. Network Boot (Minimal BOOTP) Minimal High Low High Production deployments, automated installations
2. ISO-Based None Medium Medium Medium Isolated environments, one-time installs
3. LPAR Profile None Medium High Medium Permanent configurations, few LPARs
4. Manual SMS None None Low None Troubleshooting, testing, single installs
5. Custom Script None High High Medium Custom requirements, advanced automation

Decision Guide

Choose Your Boot Method

If you need...

  • Automated, scalable deployment
    → Use Method 1: Network Boot with Minimal BOOTP (Recommended)
  • No DHCP/BOOTP at all
    → Use Method 2: ISO-Based Installation
  • Permanent LPAR configuration
    → Use Method 3: LPAR Profile Boot
  • One-time install or troubleshooting
    → Use Method 4: Manual SMS Configuration
  • Custom boot workflow
    → Use Method 5: Custom Boot Script
